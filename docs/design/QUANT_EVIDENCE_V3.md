# 量化证据 V3：时点样本、经济价值与线上校准

## 目标与边界

量化证据 V3 的目标不是承诺高收益，而是让日报与荐基只引用可解释、可复现、时点正确且具备经济价值的证据，并把证据不足明确反映到动作强度中。

系统必须始终区分五件事：

- **方向**：证据当前支持上涨、下跌、中性还是方向未知；
- **可靠性**：历史样本是否稳定、样本外是否成立；
- **覆盖度**：基金、时期和字段是否足够完整；
- **时效性**：证据在决策时是否已经可得且仍然新鲜；
- **风险守卫**：集中度、回撤等只限制动作，不得冒充收益支持。

任何一路缺失都应降级或退出，不允许由 LLM 猜测补齐。历史回测可靠不等于今天方向为正，风险数据可靠也不等于存在买入机会。

## 决策数据流

```text
交易日全基金目录
  -> 追加式 PIT 基金池快照
  -> 按当时可见成员构造同类横截面
  -> 同口径 NAV 特征与基金类型专属因子
  -> 5/20/60 日未来同类相对收益
  -> HAC + expanding walk-forward + purge/embargo + FDR
  -> 扣费后的经济显著性门槛
  -> 基金级同类百分位与因子可靠性
  -> 方向/可靠性/覆盖/时效/风险分路合成
  -> DecisionEvent 冻结当时量化快照
  -> T+N 自动结算
  -> 仅供人工复核的线上影子校准
```

## 第一批：修正证据语义并自动结算

### 证据合成

`quant_evidence.v2` 不再把多路“置信度”直接平均为一个看多分数：

- 因子证据只有在 IC 快照可用、目标基金适用且方向明确时才成为收益信号；
- 板块历史规则只有显式提供当日触发方向时才成为收益信号，否则只是历史验证；
- 组合风险固定为 `risk_guard`，只会限制动作；
- 旧 `level/score` 兼容字段严格表示**正向支持**，不再表示无方向的平均可靠性；
- 前端分别展示正向支持、可靠性、方向、覆盖、时效和风险守卫。

### 结果结算

每日任务直接读取持久化的正式 `DecisionEvent v2` 与 pending observation，自动结算：

- 日报：T+1 / T+5 / T+20；
- 荐基：T+5 / T+20 / T+60；
- 只处理正式、可审计、尚未终态的 event+horizon；
- 成熟终态不可改写，重复运行幂等，来源冲突时 fail-closed；
- 旧报告继续可见，但不进入正式统计。

## 第二批：Point-in-time 基金池与 IC v3

### 基金池快照

每个交易日晚间捕获一次当时可见的完整基金目录，经份额保守去重和分类分层后保存 1,500 个研究组合。快照和成员均为追加式不可变记录，内容哈希相同则幂等，哈希冲突拒绝写入。

发布准入至少满足：

- 源目录份额数不少于 5,000；
- 去重独立组合不少于 1,500；
- 样本为 1,200～1,500 个组合，生产目标固定为 1,500；
- 至少覆盖 4 个真实基金类别；
- 基金成立日、目录可得时间和 NAV 日期均不得晚于快照时点。

生产配置 MySQL 时禁止静默回落到 SQLite，避免服务器看似发布成功、实际只写入单机临时库。

### 防未来穿越与当前边界

每个研究锚点只选择当时已经可得且不超过 7 天的最新快照。锚点早于首个真实快照时直接退出，不能用今天的目录回填过去。

当前 v3 的真实 PIT 范围是 **membership only**：成员目录已冻结，但公开 NAV 历史没有逐笔 `available_at`，也无法证明后续修订前的原始值。因此模型必须声明 `point_in_time_scope=membership_only`、`nav_revision_pit=false`，可靠性最高为“中”，不能因为统计和经济门槛通过就称为完整 PIT 高置信。为降低穿越风险：

- 普通基金的因子截止 NAV 相对信号锚点至少滞后 1 个交易日；
- QDII 至少滞后 2 个交易日；
- 未来收益以下一交易日首个可执行 NAV 入场，持有 h 日从该 entry 计算；
- 5/20/60 日各自独立成熟，60 日标签不会拖慢 5/20 日证据更新；
- 未来只有在 NAV observation 也保存不可变 `available_at`/修订版本后，才可升级为 `nav_observation_pit` 并考虑“高”置信。

成员还必须同时满足：

- 因子回看窗口完整；
- 保守发布滞后日的因子 NAV 在允许陈旧范围内；
- 对应持有周期的下一可执行 entry/exit NAV 可验证；
- QDII 与普通基金使用不同的 NAV 陈旧容忍度。

### IC v3 统计门槛

每个 `基金类型 × 因子 × 持有周期` 独立验证。一个因子要成为 qualified，至少同时满足：

- 有效时期不少于 30；
- 5 折 expanding walk-forward 全部有效；
- 5 折中至少 4 折方向为正；
- 样本外平均 Rank IC 不低于 0.02；
- ICIR 不低于 0.20；
- HAC 95% 区间下界大于 0；
- Benjamini-Hochberg 多重检验 `q <= 0.10`；
- purge/embargo 不短于对应未来标签窗口。

整个 PIT 模型还要求有效锚点不少于 24、锚点覆盖和成员 NAV 覆盖均不低于 90%，且至少 4 个主要类别形成可用模型。任一条件不足时仍可生成诊断，但发布为诚实的 `current_survivors` 兼容快照；这类证据最高只给中等可靠性。

PIT 快照从功能上线之日起真实积累，系统不会伪造已消失基金的历史成员。因而“PIT 积累中”是正确状态，不应通过降低门槛换取虚假的高置信。

## 第三批：经济显著性、类型因子与线上影子校准

### 经济显著性

统计显著不代表扣除换手和费用后有投资价值。每个 qualified 候选还必须报告：

- 顶部/底部分位未来同类相对收益与 top-bottom spread；
- spread 的稳健标准误、t 值与置信区间；
- 顶部分位净正收益比例；
- 各分位收益及单调性；
- 换手率、盈亏平衡费率；
- 0%、0.5%、1% 成本情景下的净 spread；
- 下行 P10、最差结果与有效样本数。

上述收益标签固定为“同类横截面中位数相对总收益”，执行口径固定为下一交易日首个可执行 NAV；不能用信号当日未知 NAV 计算一个事后无法执行的收益。

经济门槛和统计门槛必须同时通过。缺少精确基金基准时，标签只能声明为“同类相对总收益”，不能冒充基金合同基准超额。

### 基金类型专属因子

共用的动量、风险调整和回撤因子继续保留，但不同类型不再机械使用同一组偏好：

| 类型 | 主要研究方向 | 必须避免的误用 |
|---|---|---|
| 主动股票 / 混合 | 中期动量、加速度、收益一致性、下行风险、回撤修复 | 不把单一短期暴涨当稳定能力 |
| 债券 | 稳定收益、下行偏差、负收益日比例、尾部损失、回撤 | 不套用股票高动量权重 |
| 指数 | 精确基准下的跟踪差异与跟踪误差 | 无精确基准时 tracking 因子必须不足 |
| QDII | 相对动量、下行风险、回撤、数据新鲜度 | 不忽略估值日与时区差异 |
| FOF | 风险调整、下行风险、修复能力、收益一致性 | 不把底层重复暴露当分散化 |

规模在没有时点历史规模序列以前只作为容量和流动性门槛，不作为预期收益因子。

类型因子只有在离线 PIT 研究通过同类统计及经济门槛、线上又能按完全相同定义计算时，才可进入日报或荐基；否则只输出 `insufficient`。

### 线上影子校准

每次正式决策冻结当时使用的因子快照 ID、模型版本、cohort、同类组、因子百分位、可靠性、适用性和数据时点。后续结果不得用更新后的因子快照回填。

`quant_evidence.v2` 将模型时点和目标基金特征时点分开冻结：

- `model_data_as_of / model_generated_at / model_published_at` 记录 IC 模型本身；
- `target_feature_as_of / target_feature_observed_at / target_feature_source` 记录该只基金特征实际算到哪一日、何时抓取、来自哪里；
- `target_nav_age_trading_days / target_return_coverage` 进入审计；普通基金超过 1 个交易日、QDII 超过 2 个交易日，或有效收益覆盖低于 90%，目标因子直接 fail-closed；
- 模型生成、发布或目标观察时间晚于决策时点（仅允许 5 分钟时钟容差）时，该事件的量化证据永久标记不可用。

校准先按决策日聚合，避免同一天多次生成报告造成伪样本扩张，再按模型版本、同类组、可靠性、因子家族、实际参与评分的因子键、方向、百分位桶与持有周期统计方向命中和相对收益。只有因子方向与最终动作同向的事件才进入该组；输出明确标记为“因子条件相关性、非因果归因”：

- 成熟决策日少于 20：`insufficient`；
- 20～59 个成熟决策日：`shadow`，只展示；
- 不少于 60 个成熟决策日且覆盖率不低于 80%：`ready_for_manual_review`；
- `auto_tuning_eligible` 永远为 `false`。

线上结果只能提示人工检查因子是否失效，不能自动修改权重、门槛、Prompt 或 Guard。

### 荐基候选覆盖

荐基候选池最多可超过一次线上 NAV 预算。系统先按质量分、板块匹配分和字段覆盖选择最多 12 只 finalists 做量化，并输出 `applicable_fund_codes` 白名单。LLM 可以描述池内其他候选，但确定性 Guard 只允许白名单内候选产生“分批买入”；未覆盖、特征陈旧或不适用的候选统一降为观察并清除金额，避免第 13 名以后的候选绕过量化。

## 真实积累周期

以下时间是固定生产口径 `rebalance_step=10` 下的**理论最短门槛**，不是到期自动保证合格：

- 20 日主周期达到 30 个统计时期约需 312 个交易日；
- 20 日主周期达到 36 个经济时期约需 372 个交易日，约 17.5 个月；
- 60 日周期达到 36 个经济时期约需 412 个交易日，约 19.5 个月；
- 首个锚点还会随净值日历有 0～9 个交易日偏移；FDR、置信区间、5 折方向一致和净成本门槛仍可能使实际时间更长。

因此上线初期出现“成员 PIT / 积累中 / 最高中等可靠性”是正确结果。历史 NAV 可用于计算收益，但不能恢复过去已经消失的基金成员，也不能证明当时看到的是修订前 NAV；系统不会用历史 NAV 冒充历史 membership 或 observation PIT。

## 生产任务

- `Factor IC Universe Capture`：交易日晚间从同一份公开目录响应同时捕获并发布成员 PIT 与 NAV 首次观测，不运行重型 NAV 回测；
- `Factor IC Refresh`：周度有界读取成员 PIT 与追加式 NAV 观测历史，执行 IC 与经济显著性研究，严格校验后发布；
- `Decision Outcome Settlement`：每日结算正式决策的成熟 T+N 结果。

三类任务均复用 Lighthouse SSH 隧道和独立 Factor IC 发布 Token；Token 只放请求头，不写命令行参数、产物或 Actions Summary。

周任务会读取最多 180 个历史成员快照，提取成员并集，再分块读取截止 `as_of` 时已真实观测的 NAV。随着成员更替，并集规模会增长；生产遥测应持续关注成员并集数、观测覆盖率、批次新鲜度和 90 分钟上限。增量读取不通过削减 PIT 样本或放宽质量门槛换速度。

## NAV 首次观测账本（2026-07-18 已实现）

- schema v17 的 `factor_ic_nav_observations` 是 SQLite/MySQL 双实现的 append-only 账本；同一值重复发布幂等，修订值追加新行，UPDATE/DELETE 均由数据库触发器拒绝。
- `first_observed_at` 只能取采集器首次看到该值的真实时间，不得从 `nav_date`、文件时间或历史接口反推。发布端拒绝未来时间以及超过 24 小时的伪回填批次。
- 历史查询按基金、净值日和显式 `as_of` 选择最早已观测版本，固定 `availability_basis=collector_first_observed_at` 与 `revision_policy=first_observed_value`；观测时间、数值、来源、批次和内容哈希共同验真。
- observation-PIT 与既有 membership-PIT 独立计算。真实时间戳覆盖不足 100%、可用因子点少于 250 或任何统计/经济门槛失败时，周任务继续保留可发布的 membership-PIT v3/v2，不会把今天拉取的历史净值回填为过去已知，也不会自动切换正式模型。

## 验收原则

- 用合成数据验证防未来穿越、purge/embargo、FDR 和成本门槛；
- 用真实公开目录做只读捕获演练，核对源份额、去重组合、样本数和分类覆盖；
- 对 SQLite、MySQL DDL、跨库迁移和追加式冲突分别测试；
- 对旧 schema、缺字段、过期快照、未知基金类型和无精确基准全部做 fail-closed 回归；
- 后端、前端、工作流 YAML、容器打包和文档必须一起验收。
