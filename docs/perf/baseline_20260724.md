# FundPilot 本地性能基线与 P0 前后对比（2026-07-24）

## 1. 结论

本次可重复的本地基准确认了一个明确热点：历史荐基报告列表虽然只向浏览器返回摘要，但旧查询仍会触碰同一行中的大 JSON payload。把摘要投影到窄表后，`/api/fund-discovery/reports` 在 50 并发下：

- P95：`2595.64 ms → 619.66 ms`，下降 `76.1%`。
- P99：`2653.42 ms → 643.27 ms`，下降 `75.8%`。
- 吞吐：`29.61 → 115.13 req/s`，提升 `288.8%`，约 `3.89×`。
- 所有层级、所有端点的请求成功率均为 `100%`。

普通日报列表在 50 并发下 P95 下降 `25.0%`，吞吐提升 `30.3%`。其他端点结果有正有负，受 Windows 本机调度、SQLite 单写者模型、响应体变化和单次短样本影响，不能外推成生产容量，也不应宣称全站一致提速。

## 2. 测量口径

基准脚本：[scripts/perf/local_api_benchmark.py](../../scripts/perf/local_api_benchmark.py)

固定条件：

- 单个本地 Uvicorn worker。
- 独立临时 SQLite；不读取或修改项目配置的真实数据库。
- 外部 provider 与后台刷新全部关闭。
- 创建一个 benchmark-only 用户并使用真实 JWT 认证。
- 预置 20 份日报，每份合成 payload 约 120 KB。
- 预置 20 份荐基报告，每份合成 payload 约 900 KB。
- 每个端点在并发 `1 / 10 / 25 / 50` 下各请求 80 次。
- 测量总延迟、TTFB、响应字节、P50/P95/P99、成功率与 RPS。
- before/after 均基于工作区 commit `54f98b0` 上的未提交实现快照；测试期间没有 Git 发布动作。

端点：

`/health`、`/api/auth/me`、`/api/investor-profile`、`/api/portfolio/holdings`、`/api/portfolio/summary`、`/api/fund-profiles`、`/api/reports`、`/api/fund-discovery/reports`。

## 6. Auth 指标的安全取舍

P0-03 的原提案同时包含 `to_thread` 与 30–60 秒 AuthPrincipal 缓存。实现保留每请求权威 DB 查询，只把同步 I/O 移出事件循环：

- 用户禁用立即生效。
- 角色/管理员权限变更立即生效。
- 密码重置和 `authVersion` 递增立即使旧 JWT 失效。
- 两 worker 不会因各自 TTL 缓存出现不一致窗口。

因此，本次没有追求原报告提出的 “Auth P95 -30%” 指标。50 并发样本中的 Auth P95 反而上升 `24.2%`，应如实视为未达该性能目标，而不是隐藏或归因成确定的回归。当前选择是用可验证的安全一致性换取较少的缓存收益；若未来要引入缓存，应先设计跨 worker 主动失效或把安全版本检查与 profile 缓存拆开。

## 7. 资源快照

- Before：进程树约 `149.57 MB`、54 threads、53 TCP connections。
- After：根 Uvicorn 进程约 `167.14 MB`、59 threads、29 TCP connections。

两个快照的进程统计范围不同（process tree 与 root process only），不能据此计算内存或线程增减百分比。TCP 数下降只可作为共享连接池方向的辅助观察，不能替代生产 `Threads_connected`、fd、握手数和 provider 压测。

## 8. 复现

在仓库根目录执行：

```powershell
python scripts/perf/local_api_benchmark.py `
  --label local-recheck `
  --output .tmp/perf/local-recheck.json `
  --requests 80 `
  --concurrency 1 10 25 50
```

脚本会创建和清理自己的临时数据库与本地服务。输出路径由调用者指定；不要指向生产数据库。

## 9. 未覆盖范围

本基准不覆盖：

- 生产 MySQL 网络 RTT、锁竞争、连接老化和 migration 执行时间。
- 两 worker 的跨进程负载分布。
- 真实 Nginx gzip、SSE buffering 与 upstream keepalive。
- 真实 DeepSeek 费用、429 brownout 和最长 180 秒预算。
- 真实东方财富/AkShare 限流、熔断与连接复用。
- 真实用户数据量、生产并发、Web Vitals 与完整 SSE 正常完成时延。

上线前仍需在 staging/生产等价环境按实施文档的部署门禁复测。
