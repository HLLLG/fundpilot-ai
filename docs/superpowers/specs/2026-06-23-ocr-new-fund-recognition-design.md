# OCR 新基金识别 + 养基宝导入下线 — 设计说明

**日期：** 2026-06-23  
**状态：** 已实现（2026-06-23）

## 背景

用户上传支付宝「全部持有」截图时，已录入基金可识别，新基金无法自动查码。养基宝总览/详情 OCR 已弃用，仅保留支付宝路径。

## 根因

1. **解析器缺口：** `指数C/A` 后缀未纳入基金名识别规则，导致如「中欧上证科创板人工智能指数C」漏识别。
2. **东财基金表加载失败（Windows）：** `subprocess.run` 未指定 `encoding=utf-8`，GBK 环境下子进程输出解码失败，名称表为空 → 新基金查码恒失败；老基金靠档案码兜底。
3. **查码失败无明确引导：** 持仓保持 `000000`，用户不知须手动搜索选码。

## 方案（一期 · 方案 A）

| 项 | 做法 |
|----|------|
| 子进程编码 | `subprocess.run(..., encoding="utf-8", errors="replace")` |
| 指数型基金 | `COMPLETE_FUND_NAME_RE` / `FUND_PRODUCT_SUFFIX_RE` 增加 `指数[A-CEH]` |
| 发行人前缀 | 补 `中航` 等常见发行人 |
| 查码失败 | **不**自动分配临时码；`fund_code_resolutions` + `holding_warnings` + 确认弹窗文字提示，引导点「搜索」从东财选取 |
| 养基宝 OCR | 删除小程序/Web 残留入口；`ocr_parser` 已为纯支付宝；清理 `yangjibao_overview` 分支与板块沉淀 source 命名 |

## 不在本期

- 编辑距离 / 分词模糊匹配（方案 B）
- 基金表本地 SQLite 缓存

## 验收

- 用户截图 fixture 解析 4/4 只基金
- `_fund_name_table()` 在 Windows 下返回 >10000 条
- 新基金查码失败时确认页显示提示，不自动填码
- pytest 全绿
