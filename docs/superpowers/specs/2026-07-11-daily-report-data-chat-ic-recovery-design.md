# 灵析日报资金证据、追问阅读与 IC 恢复设计

**日期：** 2026-07-11

**状态：** 用户已确认方案 A，待书面规格验收

**范围：** 日报中的今日/5 日主力资金、日报追问阅读体验、因子 IC 状态语义与本地验证链路

**前置规格：** `2026-07-11-daily-report-ux-redesign-design.md`

本文是前置日报改版规格的增量修订。本文第 5 章取代前置规格第 8 章的“所有桌面尺寸均使用模态抽屉”设计；其余已经落地的信息层级、基金卡片和工具区设计继续有效。

## 1. 背景与已确认事实

### 1.1 今日/5 日主力并非前端字段错误

隔离 SQLite 中的真实报告 `264415e444a246f0b92182f92eae4d0e` 使用有效交易日 `2026-07-10`。同一交易日的主题榜缓存包含 76 个板块，76 个板块均有当日主力净流入值；但报告生成时只有 3 个板块历史资金流缓存，7 只持仓中仅半导体 `BK1036` 同时得到今日与 5 日主力。

代表性对照如下：

| 基金/板块 | BK 代码 | 主题榜今日主力 | 报告今日主力 | 报告 5 日主力 |
|---|---|---:|---:|---:|
| 华夏人工智能 ETF 联接 C | BK0800 | -134.84 亿 | 缺失 | 缺失 |
| 易方达国防军工混合 C | BK0490 | 95.96 亿 | 缺失 | 缺失 |
| 华富半导体产业混合 C | BK1036 | -248.78 亿 | -248.78 亿 | -162.81 亿 |

前端 `SectorOpportunityCard` 正确读取 `today_main_force_net_yi` 和 `cumulative_5d_net_yi`。缺失发生在后端：`build_sector_fund_flow_context()` 在历史序列为空时提前返回，导致稍后才执行的主题榜“今日值兜底”永远无法运行。机会评分又用同一个 `available` 同时门控今日和 5 日字段，因此两项被一起清空。

### 1.2 5 日历史存在独立可用性问题

“今日主力”来自主题榜实时快照；“5 日主力”必须来自按交易日排列的历史资金流。历史接口可能返回空、断连或尚未被预热，现有实现还存在以下放大因素：

- 全市场约 70 个板块使用低并发顺序预热，冷启动时持仓板块不一定及时覆盖。
- 日报持仓事实与板块机会分别构建资金流上下文，可能重复请求同一板块。
- 报告机会构建有总时间预算，热门候选会与持仓板块竞争预算。
- 空结果只压缩成 `[]`，无法在报告层区分“上游无历史”与“本次网络失败”。

### 1.3 追问抽屉是桌面交互回归

当前 `ReportChatDrawer` 对所有视口统一使用模态语义：全屏遮罩、`body` 滚动锁、Tab 焦点圈闭和 `aria-modal=true`。桌面端虽然只显示右侧 420px 面板，遮罩仍覆盖日报，因此用户看得到正文却无法滚动或操作。

聊天消息区本身已有正确的独立滚动实现，问题不在 `ReportChatPanel`，而在桌面端被错误套用移动端 modal 外壳。

### 1.4 IC 任务执行过，但没有迁移到 Lighthouse

2026-07-10 的 Factor IC Refresh #2 已成功完成“生成”和“发布”步骤。但该次运行所在提交仍把发布地址硬编码为旧 CloudBase API。当前 Lighthouse 的内部发布端点返回 `503 因子 IC 发布未配置`，隔离 SQLite 的 `factor_ic_snapshots` 也为空，因此当前日报显示“无回测数据”符合实际状态。

结论：IC 计算任务曾成功执行，但成功产物发布到了旧环境，没有进入当前 Lighthouse 数据库或本地隔离 SQLite。

## 2. 目标与成功标准

### 2.1 目标

1. 只要主题榜存在同交易日数据，日报就展示真实“今日主力”。
2. 只有具备完整、日期对齐的真实历史窗口时才展示“5 日主力”，绝不以单日值冒充。
3. 优先保证持仓板块的历史资金流命中率，同时不让外部接口失败阻断日报生成。
4. 桌面端追问时可以持续阅读和滚动整份日报，聊天区保持独立滚动。
5. 将“IC 数据缺失”与“IC 回测结果弱”区分，避免把基础设施缺失解释成投资结论。
6. 在本地 SQLite、真实 DeepSeek、真实新闻/行情联网条件下完成端到端验证。

### 2.2 可验证标准

- 历史序列为空但主题榜有当日值时，`today_main_force_net_yi` 正常写入报告。
- 只有一个当日点时，`cumulative_5d_net_yi` 保持 `null`。
- “5 日主力”至少需要 5 个不重复、日期有序且不晚于报告交易日的真实交易日点。
- 缺失指标不再渲染“— 亿”，而显示“今日数据暂缺”或“5 日历史暂缺”。
- 桌面宽度不低于 1280px 时，日报正文与追问侧栏同时可操作；正文使用页面滚动，聊天使用自身滚动。
- 手机和平板继续保持遮罩、滚动锁、焦点圈闭和 Escape 关闭。
- 没有 IC 快照时显示“IC 回测数据未接入/暂不可用”，不显示“量化背书弱”。
- 本地完整验证不得连接生产 MySQL。

## 3. 范围边界

### 3.1 本次包含

- 修复资金流上下文的提前返回和今日/历史共同门控。
- 提升持仓板块历史资金流预热优先级，并减少同一报告内的重复拉取。
- 调整日报资金指标的缺失态文案。
- 将桌面追问改为 sticky sidecar，保留移动端 modal。
- 修正 IC 缺失态与弱证据态的产品文案。
- 在隔离 SQLite 中生成、发布并消费真实 IC 数据，验证日报证据链。
- 补充部署诊断与 Lighthouse IC 恢复清单。

### 3.2 本次不包含

- 不伪造、插值或估算缺失的 5 日主力资金。
- 不回写历史日报。历史报告继续保留生成当时的事实快照。
- 不改变 LLM 的投资动作词表、风险守卫阈值或调仓算法。
- 不更换东财资金流数据源，也不引入付费行情源。
- 不直接连接、查询或写入生产 MySQL。
- 不在本次任务中修改 Lighthouse `.env.production`、GitHub production secret/variable 或触发生产 IC 发布；这些属于需要单独授权的生产操作。

## 4. 主力资金数据设计

### 4.1 将“今日可用”和“历史可用”拆开

`sector_fund_flow` 不再用一个面向历史序列的 `available` 同时代表所有资金证据。上下文增加可选状态字段：

```text
today_available: bool
five_day_available: bool
history_point_count: int
today_main_force_net_yi: float | null
cumulative_5d_net_yi: float | null
```

兼容规则：

- `available` 在“今日或 5 日至少一项可用”时为 `true`，避免旧调用方把整个上下文丢弃。
- 新评分和展示逻辑使用细分状态，不再用 `available` 同时门控两项数值。
- 字段保持可选，旧报告与旧测试数据仍可读取。

### 4.2 当日值合并顺序

对每个板块执行以下流程：

```text
解析 BK 代码
  → 读取历史序列（允许为空）
  → 读取同一主题榜快照中的当日主力
  → 按日期去重并合并当日点
  → 分别计算今日状态和历史窗口状态
```

历史为空时不得提前返回。只要主题榜当日值存在，就构造带报告有效交易日的当日点，并标记：

```text
today_available = true
five_day_available = false
cumulative_5d_net_yi = null
```

如果历史中已有报告交易日点，以主题榜同请求中的当日值作为盘中/当日对齐来源；合并时按日期覆盖而非追加重复日期。

### 4.3 5 日窗口规则

5 日累计只统计以下点：

- 日期不晚于报告有效交易日。
- 日期唯一且升序。
- 主力净流入为有限数值。
- 最近窗口包含至少 5 个真实交易日点。

少于 5 点时返回 `null` 并标记 `five_day_available=false`。本次只收紧用户可见的 5 日指标口径；20 日内部指标保持现有兼容口径，避免把本次缺陷修复扩展成新的量化规则变更。

### 4.4 报告级复用与预热优先级

单份报告使用一个资金流解析结果源，按优先级处理：

1. 当前持仓板块。
2. 已选入日报的板块机会。
3. 其余主题榜后台预热。

持仓和机会构建应复用同一个 report-scoped flow map 或同一缓存结果，避免并行重复请求。优先列表使用受控并发和既有总预算；外部接口超时只使对应板块降级，不阻断报告与 DeepSeek 分析。

全市场后台预热继续作为低优先级 best-effort 任务，但不得抢占持仓报告的关键路径。

### 4.5 前端显示

`SectorOpportunityCard` 按字段分别展示：

| 状态 | 今日主力 | 5 日主力 |
|---|---|---|
| 有真实数值 | `-134.84 亿` | `-162.81 亿` |
| 当日缺失 | `今日数据暂缺` | — |
| 历史不足/失败 | — | `5 日历史暂缺` |

缺失态不附加“亿”，避免让用户误以为只是格式化错误。可以使用弱化文字和简短说明，但不新增醒目的警告卡。

## 5. 追问阅读工作区设计（方案 A）

### 5.1 响应式模式

| 视口 | 布局 | 日报正文 | 聊天 | 语义 |
|---|---|---|---|---|
| `<640px` | 底部 modal | 锁定 | 独立滚动 | `dialog` + `aria-modal` |
| `640–1279px` | 右侧 420px modal | 锁定 | 独立滚动 | `dialog` + `aria-modal` |
| `>=1280px` | 正文 + 360–400px sticky sidecar | 页面正常滚动 | 独立滚动 | `complementary` |

1280px 以下不强制双栏，避免正文被压缩到难以阅读的宽度。

### 5.2 组件结构

`ReportPanel` 外层增加无动画的 `report-workspace`，包含：

```text
report-workspace
  ├─ report document section
  └─ ReportChatDrawer / sidecar
```

打开桌面 sidecar 时，工作区使用：

```text
minmax(0, 1fr) minmax(360px, 400px)
```

聊天侧栏使用 `position: sticky`，顶部避开导航，明确设置可用视口高度，使 `ReportChatPanel` 的内部 `overflow-y-auto` 生效。日报正文继续使用 body/page scroll，不引入第二个正文滚动容器。

### 5.3 单实例与 SSE 生命周期

桌面和移动端必须复用同一个 `ReportChatPanel` 子树，通过响应式外壳改变布局和语义，不能同时渲染两份聊天组件。否则会导致：

- 对话历史重复请求。
- 断点切换时活动 SSE 被卸载中止。
- 同一问题产生重复 token 流或状态不一致。

关闭面板时仍条件卸载聊天组件，并保留现有 AbortController 清理，确保活动 SSE 及时中止。

### 5.4 焦点和键盘

- 桌面：无全屏遮罩、无 body lock、无 Tab trap；Escape 和关闭按钮均可关闭并恢复触发按钮焦点。
- 手机/平板：保留遮罩点击关闭、body lock、焦点圈闭、Escape 和焦点恢复。
- 触发按钮增加 `aria-expanded` 与 `aria-controls`。
- sidecar 打开后，悬浮触发按钮从键盘和指针顺序中移除，关闭后恢复。
- `ReportSummaryHero` 使用容器宽度决定内部排版，sidecar 打开导致主列变窄时保持纵向摘要，避免小列强行套用 viewport `lg` 双栏。

## 6. IC 状态与恢复设计

### 6.1 产品语义

量化证据存在三种状态：

1. **未接入/不可用：** 没有 IC 快照、状态接口失败或快照字段无效。
2. **已接入但过期：** 有快照，但超过配置的新鲜度阈值。
3. **已接入且可评估：** 有有效快照，才允许显示因子“高/中/弱”等可靠度。

当所有因子依据均为“无回测数据”时，日报显示：

```text
量化回测未接入
当前建议主要依据持仓风险、行情与新闻；IC 不参与本次结论。
```

只有存在有效 IC 快照且因子结果确实不足时，才使用“量化背书弱”。数据缺失不能被包装成负面量化结论。

### 6.2 本地真实验证

使用与 GitHub workflow 相同的回测参数，在联网环境中运行 `run_factor_ic.py`，得到经过 schema 校验的 `summary.json`。随后通过本地专用发布 Token 调用内部发布服务，将快照写入隔离 SQLite 的 `factor_ic_snapshots`。

验证顺序：

```text
真实市场数据生成 IC summary
  → 校验 schema/日期/样本数
  → 发布到隔离 SQLite
  → /api/diagnostics/factor-ic-status 可用
  → 生成真实 DeepSeek 日报
  → 日报 factor_reliability 不再是“无回测数据”
```

本地验证仅使用任务专用 SQLite 文件，不读取或写入生产 MySQL。

### 6.3 Lighthouse 恢复清单

生产恢复需要独立授权，且必须满足以下前置条件：

1. 正式域名 HTTPS 与证书正常。
2. Lighthouse `.env.production` 配置 `FUND_AI_FACTOR_IC_PUBLISH_TOKEN`。
3. GitHub `production` Environment secret 配置相同的 `FACTOR_IC_PUBLISH_TOKEN`。
4. `FACTOR_IC_PUBLISH_URL` 指向正式 HTTPS 的 `/api/internal/factor-ic-snapshots`。
5. `FACTOR_IC_REFRESH_ENABLED=true`。
6. 手动触发一次 workflow，并通过应用状态接口确认 run date、样本数与非 stale 状态。

不得把发布 URL 降级为 HTTP，也不得为了验收直接开放或连接生产 MySQL。

## 7. 错误处理与降级

- 主题榜有今日值、历史接口失败：展示今日值；5 日显示历史暂缺。
- 主题榜和历史都失败：资金两项分别显示诚实缺失态，报告继续生成。
- 资金日期与报告交易日不一致：不展示为“今日”，也不参与量价背离判断。
- 历史不足 5 点：不计算 5 日累计。
- IC 无快照：显示未接入，不参与量化可靠度结论。
- IC 过期：显示日期与过期提示，不将其升级为强证据。
- 聊天请求失败：只在 sidecar/modal 内显示，不影响日报滚动。
- 断点变化：保持单一聊天实例和当前对话，不重新请求历史。

## 8. 测试策略

### 8.1 后端 TDD

- 历史 `[]`、主题榜有今日值：今日保留，5 日为空。
- 历史 4 点加同日实时点：去重后满足 5 点才计算累计。
- 不足 5 点、重复日期、未来日期、非有限值均不得生成 5 日累计。
- 今日与历史可用性分别进入 `sector_opportunity`，不再共同清空。
- 单份报告持仓与机会复用 flow 结果，关键板块不会重复拉取。
- 外部资金接口超时不阻断报告事实构建。
- 无 IC、过期 IC、有效 IC 三种状态映射正确。
- 本地 IC 发布写入隔离 SQLite，并可被 `factor_confidence` 消费。

### 8.2 前端 TDD

- 今日/5 日有值时展示数字。
- 今日缺失显示“今日数据暂缺”；5 日缺失显示“5 日历史暂缺”；均不显示“— 亿”。
- 桌面 `matchMedia >=1280px` 时为 `complementary`，body 可滚动且无焦点圈闭。
- 移动端继续为 modal，保留滚动锁、焦点圈闭、Escape 和遮罩关闭。
- 正文与 sidecar 是同一工作区的 sibling，聊天不会覆盖正文。
- 断点切换时历史只加载一次，活动 SSE 不因布局切换中止。
- 关闭任一模式都会中止活动 SSE 并恢复触发按钮焦点。
- IC 缺失显示“未接入”，有效但低可靠度才显示“背书弱”。

## 9. 发布与验收

1. 先提交本规格并由用户验收。
2. 编写可执行实施计划，按 TDD 分成资金流、追问工作区、IC 语义/验证三个独立子任务。
3. 子任务完成后进行规格符合性审查和代码质量审查。
4. 全量自动化通过后，在隔离 SQLite 上生成一份全新真实日报。
5. 由用户在本地真实页面验收。
6. 生产 IC 恢复仍保持未执行，除非用户另行明确授权 Lighthouse/GitHub 环境变更。

本规格不得被解释为连接或修改生产 MySQL 的授权。
