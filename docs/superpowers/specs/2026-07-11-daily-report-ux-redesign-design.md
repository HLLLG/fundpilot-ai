# 灵析日报页面全面优化设计

**日期：** 2026-07-11

**状态：** 已完成用户逐段确认，待书面规格复核

**范围：** 整个「日报」Tab，包括生成设置、报告正文、追问助手、扩展工具，以及已确认影响日报可信度的诊断指标缺陷

## 1. 背景与现状

日报后端已经能输出结构化动作、仓位变化、新闻、决策路径、板块依据、基金依据、量化置信和校验备注。现有前端为了保证可追溯性，几乎把全部字段默认平铺在每只基金卡片中，导致关键行动与专业证据处于同一视觉层级。

基于项目文档、相关源码、近期提交、用户截图及生产页面实测，确认以下问题：

- 桌面端 7 只持仓的决策区约 4448px 高，单张卡片约 540–708px；整个页面约 6330px。
- 390×844 手机视口下页面约 9154px，报告主体约 7870px。
- 默认平铺的 `points`、新闻、决策路径、证据网格、校验备注与风险存在明显语义重复。
- 追问助手桌面端固定接近一屏高度并长期占用约 26rem 宽度；无对话时产生大面积空白。
- 完整 AI 角色 Prompt 在已有报告时仍占据较高位置，推迟用户看到结果的时间。
- “暂无明确利好 / 暂无明确利空”仍以绿色或红色卡片渲染，制造无效信号。
- 历史或边界报告会泄漏 `opportunity absent`、`daily_return pending`、`momentum` 等内部术语。
- 生产报告出现近 1 年收益 8220.94%、最大回撤 -160.53% 等不可信值。根因是 `_parse_return_frame` 把累计收益率百分比序列直接当净值序列做比值和回撤计算。
- `RiskControls` 高级设置中的 `prefer_dca` 复选框缺少可见标签。
- 现有测试主要验证结构化字段是否出现，未覆盖默认折叠、异常值、占位新闻、响应式阅读长度和抽屉行为。

## 2. 目标与成功标准

### 2.1 目标

1. 小白在 10 秒内能回答：今天要不要动、先动哪只、动多少、什么条件下再行动。
2. 不删除有价值的专业证据，而是将其移入按需展开层。
3. 报告正文获得完整阅读宽度，追问功能仍易于发现。
4. 桌面端和手机端使用同一信息顺序，无横向滚动。
5. 修复会直接损害报告可信度的数据和文案缺陷。
6. 保持日报生成、历史报告、Markdown 导出和 SSE 追问协议兼容。

### 2.2 可验证标准

- 默认首屏能看到组合结论以及至少第一只“需要处理”的基金。
- “需要处理”卡片的默认内容只包含行动必需信息；“继续观察”卡片默认为紧凑行。
- 典型 7 只基金报告的默认页面高度相较当前生产页面至少降低 50%。
- 桌面端不再因空追问区损失正文宽度；手机端 390px 视口无横向溢出。
- 无意义的利好/利空占位语不产生彩色卡片。
- 前端不显示最大回撤不在 `[-100, 0]` 或近 1 年收益不在 `[-100, 1000]` 的异常指标。
- 新生成报告和历史报告都不应直接显示已知内部枚举或字段名。
- 关键键盘操作、焦点恢复和折叠状态满足可访问性要求。

## 3. 不在本次范围内

- 不改变 LLM 的投资决策语义、动作词表或风险守卫阈值。
- 不新增日报 API 主结构，不迁移历史报告表。
- 不重写流式生成、后台任务、报告导出或聊天服务。
- 不持久化基金卡片展开状态到数据库或浏览器长期存储。
- 不对日报以外页面做无关的视觉重构。
- 除已确认的基金诊断指标计算错误外，不扩展到新的量化模型或数据源研究。

## 4. 已选方案

用户从三套方向中选择并确认 **A：行动优先单列**。

其余备选方案为：

- 主从双栏：保留右侧风险与追问，但继续牺牲正文宽度。
- 决策清单表：扫描效率高，但更像专业工作台，不符合本次“小白优先”的核心目标。

行动优先单列按紧急程度分组，先给结论，再解释原因，最后保留专业证据。追问助手改为按需抽屉，不再常驻右栏。

## 5. 页面结构

### 5.1 页面状态

日报 Tab 存在两种生成设置状态：

1. **尚无日报：** 保持完整生成设置，包括快速/深度模式、投资预设、角色设定、生成按钮和高级设置。
2. **已有日报：** 自动进入阅读模式。生成设置压缩为一行，显示当前模式、投资风格、浮亏线、集中度，并提供“调整设置”和“重新生成”。展开“调整设置”后复用原完整控件。

新报告生成完成后应自动回到压缩状态。用户在当前页面主动展开设置时保持展开，直到生成新报告或离开页面。

### 5.2 阅读顺序

页面从上到下排列：

1. 交易时段与数据日期。
2. 压缩后的生成设置。
3. 今日组合结论。
4. “需要处理”的基金。
5. “继续观察”的基金。
6. 更多内容与工具。
7. 持续可发现但不占正文宽度的追问入口。

### 5.3 今日组合结论

组合结论区只回答总体问题：

- 报告标题或压缩后的行动结论。
- 一段精简摘要。
- 组合收益、风险等级、需要处理数量。
- Markdown 导出入口。

模型提供方名称移入报告元信息或“专业依据”，不再作为首屏状态徽标。原有组合级 `recommendations` 不再逐条渲染成独立大卡；去重后用于补充组合摘要，剩余内容进入“组合说明”折叠区。

## 6. 基金分组与排序

前端只做展示分组，不改变后端动作：

- **需要处理：** 动作 tone 为加仓、减仓、风控复核、大幅减仓或清仓，或者存在非零 `suggested_position_change_percent`。
- **继续观察：** 观察、暂停追涨和无法识别的兼容动作。暂停追涨排在普通观察前面。

分组内保持后端原始顺序，避免前端另造投资优先级。若旧报告只有字符串 `recommendations`，继续使用现有兼容解析逻辑，再应用同一分组规则。

## 7. 三层信息结构

### 7.1 默认显示：行动必需信息

“需要处理”基金默认显示：

- 基金名称，代码弱化显示。
- 最终动作。
- 把 `confidence` 映射成更易懂的“参考度：高 / 中 / 有限”。
- 一句话核心原因。
- 建议金额或仓位变化；没有结构化金额时不补造。
- 下一交易日条件化预案。
- 最重要的一条风险。

“继续观察”基金默认只显示一行：基金名称、动作、最关键原因和展开按钮。

核心原因优先级：结构化仓位变化依据 → `amount_note` → 第一条非系统改写说明的 `points` → 板块方向摘要 → 安全兜底文案。下一交易日预案从包含“下一交易日 / 下交易日 / 开盘”等明确条件的 `points` 中提取；未找到时不伪造。

### 7.2 第一层展开：“为什么这样建议”

显示：

- 去重后的 2–3 条关键理由。
- 真实、非占位的利好或利空新闻。
- 其余结构化风险。

过滤空值以及“暂无明确利好”“暂无明确利空”“暂无利好”“暂无利空”“无”等占位语。若一类新闻过滤后为空，不渲染该容器。

### 7.3 第二层展开：“专业依据”

显示：

- 最新净值与日期、基金类型、管理费等元数据。
- 板块方向、显著回测证据与量化综合置信。
- 决策路径。
- 板块依据、基金依据和校验备注。

专业依据保留现有可追溯能力，但不参与默认阅读流。极端动作仍先经过现有 `ExtremeActionGate` 二次确认，再显示行动摘要和后续证据。

展开状态只属于当前挂载的报告组件；切换报告时重置，避免把上一份报告的展开状态带入新报告。

## 8. 追问助手

移除报告正文中的常驻双栏布局和固定最小高度。

- 桌面端：点击“追问这份日报”后，从右侧打开约 420px 抽屉。
- 手机端：打开全屏面板，并避开现有底部导航安全区。
- 抽屉内部复用现有 `ReportChatPanel` 的历史、快速/深度模式、推荐问题、流式回答和导出能力。
- ESC、关闭按钮和遮罩均可关闭；关闭后焦点返回触发按钮。
- 抽屉打开时锁定背景滚动，关闭后保持用户原来的报告阅读位置。
- 请求或历史加载失败继续在聊天面板内部显示，不影响报告正文。

建议新增 `ReportChatDrawer` 负责弹层行为；`ReportChatPanel` 只负责聊天内容，并增加明确的 `variant="drawer"` 表面模式，避免继续叠加互斥布尔属性。

## 9. 更多内容与工具

当前连续五个大号折叠条合并为“更多内容与工具”入口网格：

- 主题要闻摘要。
- 板块轮动参考。
- 调仓示意模拟。
- 建议复盘与投研诊断。

入口展示简短说明或状态，不展示空白大容器。模块仍在用户打开后挂载，保持现有懒加载和错误隔离。诊断区继续属于辅助信息，不进入默认决策流。

## 10. 组件边界

为避免继续扩大当前约 477 行的 `ReportPanel.tsx`，按职责拆分：

- `ReportPanel`：报告状态编排与顶层数据派生。
- `ReportSummaryHero`：组合结论、关键指标和导出。
- `ReportRecommendationList`：分组、空态和基金列表。
- `FundRecommendationCard`：三层卡片与极端动作门禁。
- `ReportDetailsHub`：更多内容与工具入口。
- `ReportChatDrawer`：桌面抽屉、手机全屏、焦点和滚动管理。

纯数据逻辑提取为以下可单测 helper：

- `groupFundRecommendations`。
- `selectPrimaryReason`。
- `selectNextTradingPlan`。
- `meaningfulNewsLines`。
- `safeDiagnosticMetrics`。

`RiskControls` 增加受明确 prop 驱动的阅读模式，不自行读取报告全局状态。`Dashboard` 根据是否存在已完成报告传入状态。

## 11. 数据流与兼容

主数据流保持不变：

```text
Report API / 流式完成事件
  → Dashboard 持有 report
  → ReportPanel 派生摘要、分组、卡片层级
  → 用户按需展开证据或打开聊天抽屉
  → ReportChatPanel 继续调用原有 chat API
```

兼容规则：

- `fund_recommendations` 为空时继续解析旧版字符串建议。
- 缺少仓位、置信或证据字段时隐藏对应 UI，不显示 `undefined` 或占位卡。
- 旧报告中的内部术语由前端翻译兜底；新报告同时在后端 guard 层规范化。
- 本次不改变 `Report` TypeScript 类型的必填字段，也不要求数据库迁移。

## 12. 已确认数据缺陷修复

### 12.1 根因

AkShare“累计收益率走势”返回的是累计收益百分比序列。当前 `_parse_return_frame` 直接计算 `end / start - 1`，等价于把百分比当净值，因此会把约 1% 到 83% 的变化错误放大为约 8200%，并可能得到低于 -100% 的最大回撤。

### 12.2 正确口径

对每个累计收益百分比 `r` 转换为增长指数：

```text
growth_index = 1 + r / 100
```

然后在当前约 260 个样本窗口上计算：

```text
return_percent = (last_index / first_index - 1) × 100
drawdown_percent = min(current_index / running_peak - 1) × 100
```

非有限值或小于等于 0 的增长指数不参与计算；有效点不足两个时不返回收益和回撤。

### 12.3 缓存与旧报告

- `fund_diagnostics_cache` 版本从 `v1` 升级，部署后不继续命中旧错误缓存。
- 新数据在后端按正确口径生成。
- 已落库历史报告不回写；前端 `safeDiagnosticMetrics` 对最大回撤不在 `[-100, 0]`、近 1 年收益不在 `[-100, 1000]` 或数值非有限的数据隐藏，并在专业区显示“指标数据异常，已隐藏”，不让错误数字进入默认阅读层。

## 13. 文案清洗

后端 `decision_guard_shared` 和日报专属替换表补齐已知枚举；前端 `translateEvidenceText` 保留历史报告兜底。至少覆盖：

- `opportunity absent / present` → “当前不构成机会 / 当前构成机会”。
- `daily_return pending` → “当日涨跌待确认”。
- `momentum / setup` → “顺势 / 蓄势”。
- 已知内部字段名继续转换为现有人话化名称。

转换应使用有边界的正则或结构化值，避免把基金名称或正常英文新闻标题误替换。

## 14. 表单与可访问性

- 为 `prefer_dca` 恢复“偏好定投”可见标签，并确保点击文字可以切换复选框。
- 所有折叠按钮包含 `aria-expanded` 和稳定的 `aria-controls`。
- 抽屉有可识别标题、关闭按钮、初始焦点、焦点约束与焦点恢复。
- 手机端主要点击目标不小于 44px。
- 动画遵守 `prefers-reduced-motion`。
- 颜色不作为动作和风险的唯一表达，始终保留文字。

## 15. 错误处理

- 报告字段缺失时优先省略对应 UI，不用空卡占位。
- 新闻过滤后为空时不显示利好/利空区。
- 专业指标异常时隐藏具体值并给出诚实说明。
- 聊天失败局部显示，关闭抽屉后报告仍可正常阅读。
- 更多工具中的单个请求失败不得阻断其他工具或正文。
- Markdown 导出失败继续使用按钮禁用和既有异常处理，不改变本次页面状态。
