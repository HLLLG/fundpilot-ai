# Factor IC 排行榜获取容错设计

**状态：** 已实现并验证
**日期：** 2026-07-10
**关联实现：** `.github/workflows/factor-ic-refresh.yml`、`apps/api/scripts/run_factor_ic.py`、`apps/api/app/services/akshare_subprocess.py`

## 1. 问题与根因

`Factor IC Refresh` 在 GitHub-hosted Runner 获取开放式基金排行榜时，子进程运行满 60 秒后超时。现有 `fetch_open_fund_rank()` 将超时折叠为 `None`，runner 再将其解释为空基金池，最终生成 `available=false`、`universe_size=0` 的报告并退出 1。

当前固定依赖 `akshare==1.18.64`。该版本的 `fund_open_fund_rank_em(symbol="全部")` 请求东方财富排行榜时固定传入 `pn=30000`，一次获取全部基金，且内部 `requests.get()` 没有连接或读取超时。Factor IC 的生产口径只需要排名前 500 条，因此全量请求同时放大了响应体、网络波动和无界等待风险。

## 2. 目标与非目标

### 目标

- 保持“东方财富开放式基金排行榜、近一年收益降序”的现有选样口径。
- 只请求本次回测需要的前 500 条，避免下载全部基金。
- 为单次 HTTP 请求设置连接和读取超时，并对瞬时失败执行最多 3 次有界重试。
- 排行榜最终不可用时，明确报告数据源失败，不再伪装成“有效基金数 0”。
- 所有自动化测试保持离线，不调用 AkShare、东方财富、GitHub、CloudBase 或生产 API。

### 非目标

- 不引入第二排行榜数据源、代理、自托管 Runner 或生产 API 中转。
- 不提交静态基金池，也不改变 `sampled 500→300` 的生产参数。
- 不修改 NAV 获取、IC 计算、质量门槛、发布协议或数据库结构。

## 3. 方案选择

采用“同源限量直取 + 有界重试”。子进程仍负责隔离外部数据依赖，但不再调用会强制拉取全部数据的 AkShare 高层函数；子进程改为请求该函数使用的同一个东方财富 `rankhandler.aspx` 端点，并把 `pn` 设置为调用方所需的上限（生产为 500）。

未采用的方案：

- 单纯把子进程超时提高到 180 秒：仍然下载全部基金，内部 HTTP 请求仍无界。
- 使用最近成功的静态基金池：会引入时效性、更新责任和快照治理，超出本次故障范围。

## 4. 数据流与组件边界

1. `run_factor_ic.py` 按原逻辑调用 `_default_fetch_rank(500)`。
2. `fetch_open_fund_rank(limit=500)` 构造只请求前 500 条的子进程脚本。
3. 子进程对同一东方财富排行榜端点执行最多 3 次请求：每次都有连接/读取超时，失败后按短退避间隔重试。
4. 子进程只输出 JSON，不输出请求 URL、响应正文或任何环境变量。
5. 父进程校验 JSON 结构，返回标准化排行榜行。
6. 三次均失败或返回空数据时，父进程返回失败；runner 在下载任何 NAV 前终止，并输出可定位的数据源错误。

重试只覆盖连接错误、读取超时、5xx/限流响应、解析失败和空响应。确定性的本地参数错误不重试。

## 5. 失败语义

- 排行榜成功：行为与当前实现一致，继续分层抽样和 NAV 获取。
- 瞬时失败后恢复：记录不含敏感信息的 warning，继续执行。
- 全部尝试失败：runner 抛出明确的 `FactorIcRankUnavailable`，CLI 输出“开放式基金排行榜获取失败”，退出非零；不写出可被误判为真实回测结果的 `summary.json`。
- 排行榜返回不足 500 条但非空：保留真实返回结果，后续仍由现有有效基金数与发布质量门槛判定。

## 6. 测试与验收

按 TDD 先增加失败测试，再实现最小修复：

- 限量契约：生产 `limit=500` 时，请求参数不得退化为 `pn=30000`。
- 超时契约：外部请求必须带连接和读取超时。
- 重试契约：前两次瞬时失败、第三次成功时返回数据；三次失败后停止，不无限重试。
- 失败语义：排行榜最终不可用时，runner 在 NAV 获取前失败，错误信息指向排行榜数据源，且不生成误导性 `summary.json`。
- 回归：现有 Factor IC、发布端点、工作流静态契约和后端全量测试全部通过。
- 在线诊断只用于人工确认同端点的限量响应，不进入自动化测试。

生产验收：修复推送后手动重跑 `Factor IC Refresh`，确认排行榜阶段不再出现 60 秒子进程超时，最终 Actions Summary 包含有效基金数、回测期数和四个因子结果。

## 7. 实现验证

- 排行榜限量、三次重试、runner 提前失败与工作流契约：20 passed。
- 后端全量测试：656 passed；仅保留既有 Starlette TestClient 弃用警告。
- 在线只读诊断：同源限量请求返回 500 条标准化基金记录；未调用生产发布 API，未读取发布 Token。
